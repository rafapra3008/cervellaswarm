## 9. RISCHI & MITIGAZIONI

### 9.1 Rischi Percepiti

**Rischio:** "Troppo complesso, voglio solo 1 AI veloce"
**Mitigazione:**
- Quick start semplice come single agent
- "You don't see 16 agents, you see verified output"
- Default mode nasconde complessita

**Rischio:** "Guardiani rallentano il processo"
**Mitigazione:**
- Mostrare metriche: +30% tempo, -70% rework
- "Better to verify before merge than debug after deploy"
- Speed mode (skip guardians) per prototyping

**Rischio:** "Open source = no support"
**Mitigazione:**
- Active Discord/Discussions
- Fast issue response commitment
- Enterprise support option (future)

**Rischio:** "Solo per Claude? Vendor lock-in"
**Mitigazione:**
- "Claude-native by choice, not limitation"
- Roadmap: other LLM support
- Philosophy: start with one model, expand later

### 9.2 Obiezioni Comuni & Risposte

**"Perche non usare un LLM piu grande invece di multi-agent?"**
```
Great question. Bigger models are smart, but:
1. Specialization > generalization for complex tasks
2. Review requires different context than creation
3. Team structure mirrors real workflows (proven model)
4. Multiple models = diverse verification
```

**"AI non puo verificare AI, serve umano"**
```
Agree partially! That's why:
1. Guardians verify, you approve (human-in-loop at top)
2. Multiple specialized verifiers > single reviewer
3. It's not about replacing human judgment, it's about augmenting it
4. Like code review: senior dev reviews, you merge
```

**"Sembra overkill per piccoli progetti"**
```
Fair point. CervellaSwarm shines when:
1. Production code (quality matters)
2. Team projects (consistency matters)
3. Complex features (specialization matters)

For quick scripts? Single AI is fine.
For code that ships? Guardian review helps.
```
