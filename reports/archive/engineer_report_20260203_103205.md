## 3. DIPENDENZE TRA FILE

### 3.1 Gerarchia Lettura

```
INIZIO SESSIONE:
1. PROMPT_RIPRESA_MASTER.md  → Identifica progetto
2. PROMPT_RIPRESA_{nome}.md  → Context ultimo lavoro
3. stato.md                  → Solo se serve approfondimento

DURANTE SESSIONE:
- roadmaps/                  → Piano corrente
- decisioni/                 → Decisioni passate
- CONFIG.md                  → Convenzioni codice

FINE SESSIONE:
1. Aggiorna PROMPT_RIPRESA_{nome}.md  (sempre!)
2. Aggiorna stato.md                  (se significativo)
3. Aggiorna PROMPT_RIPRESA_MASTER.md  (TL;DR + data)
```

### 3.2 Dipendenze Logiche

```
PROMPT_RIPRESA_{nome}.md dipende da:
├── stato.md (riassunto last status)
├── roadmaps/*.md (next steps)
└── decisioni/*.md (decisioni recenti)

stato.md dipende da:
├── CONFIG.md (stack info)
├── roadmaps/*.md (piano generale)
└── decisioni/*.md (tutte decisioni storiche)

CONFIG.md è STANDALONE:
└── Info statiche progetto
```

### 3.3 NORD.md - File Speciale

```
Path: ROOT PROGETTO (es: ~/Developer/miracollogeminifocus/NORD.md)
NON: .sncp/progetti/miracollo/NORD.md

NORD.md = Visione e direzione (legge della bussola)
- MAI dentro .sncp/
- Sempre nella root del repository
- Link da stato.md se serve
```

## 10. DECISIONI ARCHITETTURALI

### 10.1 Perché sncp-onboard invece di estendere sncp-init?

**DECISIONE:** Nuovo script `sncp-onboard`

**MOTIVAZIONI:**
1. sncp-init è usato in produzione → rischio regressione
2. Naming più chiaro (onboard vs init)
3. Può chiamare sncp-init internamente (composition)
4. Rollback facile se problemi

**ALTERNATIVE SCARTATE:**
- Estendere sncp-init → Troppo rischioso
- Flag --full in sncp-init → Confusing

### 10.2 Perché NORD.md in repo root?

**DECISIONE:** NORD.md va in `~/Developer/{nome}/NORD.md`

**MOTIVAZIONI:**
1. Coerenza con CervellaSwarm (ha NORD.md in root)
2. File "sacro" visibile nel repo
3. .sncp/ è memoria operativa, NORD è visione
4. Git-tracked (parte del codice)

**ALTERNATIVE SCARTATE:**
- .sncp/progetti/{nome}/NORD.md → Hidden, non git-tracked

### 10.3 Template embedded vs file esterni?

**DECISIONE:** Template embedded in script (come sncp-init)

**MOTIVAZIONI:**
1. Single file execution (no dipendenze)
2. Coerenza con sncp-init esistente
3. Facile distribuzione
4. Template semplici (non complessi)

**FUTURE:** Se template > 100 righe → refactor a file separati
